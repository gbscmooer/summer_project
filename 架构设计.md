# 校园二手交易平台 —— 架构设计文档

> 课程：23级2026暑期学校《项目实训》
> 项目代号：CampusTrade（校园淘）
> 设计目标：用一个真实场景串起课程全部技术点，便于 AI 辅助编码 + 答辩讲解

---

## 一、为什么选这个题目

校园二手交易是学生最熟悉的场景，功能边界清晰、业务链路完整（发布 → 搜索 → 下单 → 通知），
能**自然地**用上课程要求的每一项技术，而不是为了用技术硬凑功能。答辩时每个技术点都能讲出"为什么用它"。

## 四、微服务拆分

| 服务 | 端口 | 职责 | 关键技术 |
|------|------|------|---------|
| `campus-gateway` | 8080 | 统一入口、路由、JWT 校验、限流 | Spring Cloud Gateway |
| `campus-user` | 8081 | 注册、登录、个人中心、Token 签发 | SpringBoot + JWT + MyBatis-Plus |
| `campus-product` | 8082 | 商品发布、详情、搜索、缓存 | Redis + Elasticsearch |
| `campus-order` | 8083 | 下单、订单查询、发通知 | OpenFeign + RabbitMQ |
| `campus-common` | - | 公共模块（非服务） | Result/异常/工具/JWT 工具 |

**拆分理由（答辩用）：**
- 按**业务领域**拆分（用户 / 商品 / 订单），符合微服务单一职责。
- 三个服务之间有真实调用关系：**订单服务下单时，要调用用户服务查收货人、调用商品服务扣库存** → 天然用上 OpenFeign。
- 没有过度拆分（比如没把"评论""收藏"单独拆服务），控制在 20 天能做完的范围。

## 六、数据库设计（MySQL）

> 采用**库内分表、逻辑独立**：每个服务只操作自己的表，不跨服务直接 join（跨服务用 OpenFeign）。

| password | varchar(100) | 密码（加密存储） |
| nickname | varchar(50) | 昵称 |

| 字段 | 类型 | 说明 |

| 字段 | 类型 | 说明 |

## 七、核心技术方案

### 7.1 统一响应 & 异常（Day2）

```java
// 统一响应
public class Result<T> {
    private Integer code;   // 200 成功，其它失败
    private String message;
    private T data;
    public static <T> Result<T> success(T data) { ... }
    public static <T> Result<T> error(String msg) { ... }
}

// 统一异常
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(BizException.class)
    public Result<?> handle(BizException e) { return Result.error(e.getMessage()); }
}
```

### 7.2 JWT 鉴权（Day2 + Day3）

- 登录成功后 `user-service` 用 `JwtUtil` 签发 Token（含 userId）。
- 前端把 Token 放在请求头 `Authorization`。
- **网关统一拦截**：`AuthGlobalFilter` 校验 Token，解析出 userId 透传给下游服务（放在请求头 `X-User-Id`）。
- 白名单：登录/注册/商品列表查询接口放行。

### 7.3 Redis 缓存策略（Day4）

| 场景 | 方案 |
|------|------|
| **缓存预热** | 项目启动时 `CommandLineRunner` 把热门商品（浏览量 Top N）加载进 Redis |
| **缓存穿透** | 查询不存在的商品时，缓存空值 `null` 并设置短过期（如 60s）；或布隆过滤器 |
| **缓存雪崩** | 缓存过期时间加**随机值**（如 30min ± 5min），避免同时失效 |
| 商品详情 | `product:detail:{id}` 缓存，读多写少，过期 30min |
| 浏览量 | Redis `INCR` 计数，定时刷回 MySQL |

### 7.4 Elasticsearch 搜索（Day7）

- 商品发布/修改时，**双写**：写 MySQL + 同步写 ES（或通过 MQ 异步同步）。
- 搜索接口走 ES：支持标题/描述分词搜索、按价格/时间排序、分类过滤。
- `product-service` 集成 `spring-data-elasticsearch`，定义 `ProductDocument` + `ProductRepository`。

### 7.5 RabbitMQ 消息队列（Day6）

| 场景 | 方案 |
|------|------|
| **下单通知** | 下单成功后，发消息到 MQ，由消费者异步给卖家发"你的商品被拍下"通知 |
| **流量削峰** | 设计一个"限时抢购"功能：抢购请求先入 MQ 排队，后端匀速消费下单，避免瞬时高并发压垮数据库 |
| **ES 同步** | 商品变更消息 → ES 消费者更新索引（解耦） |

交换机/队列设计：
- `order.exchange`（topic）→ `order.notify.queue`（通知）、`order.es.queue`（同步ES）

### 7.6 OpenFeign 服务调用（Day3）

```java
@FeignClient(name = "campus-product")
public interface ProductFeignClient {
    @GetMapping("/product/{id}")
    Result<ProductVO> getProduct(@PathVariable Long id);
}
```
订单服务下单时调用 → 校验商品是否存在、是否在售、扣库存。

### 7.7 Jmeter 压测（Day6）

- 对"下单接口" / "商品搜索接口"做并发压测（如 500 并发、循环 10 次）。
- 输出**聚合报告**：吞吐量(TPS)、平均响应时间、错误率。
- 对比"加 Redis 缓存前后""加 MQ 削峰前后"的性能差异 → 答辩亮点。

## 八、部署架构（Day1 + Day4）

### 8.1 Docker Compose 编排

```yaml
# 中间件一键启动
services:
  nacos:       # 注册中心 + 配置中心   8848
  mysql:       # 数据库               3306
  redis:       # 缓存                 6379
  elasticsearch: # 搜索               9200
  rabbitmq:    # 消息队列             5672 / 15672
  # 业务服务
  campus-gateway:  8080
  campus-user:     8081
  campus-product:  8082
  campus-order:    8083
  nginx:           80    # 前端 + 反向代理
```

### 8.2 部署流程

1. 云服务器（公网）安装 Docker + Docker Compose。
2. 各服务打成镜像（`Dockerfile` + Maven `spring-boot:build-image` 或手写）。
3. `docker-compose up -d` 一键拉起全部组件。
4. Nginx 配置：`/` → 前端静态资源，`/api` → 转发到网关，多实例时做负载均衡。

## 十一、答辩要点清单（提前准备）

每个技术点准备一句"为什么用它"，这是答辩拿分关键：

- **为什么拆微服务？** —— 业务解耦、独立部署、便于扩展（结合订单调用用户/商品讲）。
- **CAP 怎么选？** —— Nacos 默认 AP（保证可用性），注册中心场景下可用性优先。
- **缓存三兄弟怎么解决？** —— 穿透（空值/布隆）、雪崩（随机过期）、击穿（互斥锁）。
- **MQ 解决了什么？** —— 异步解耦（通知）、削峰（抢购排队）。
- **ES 和 MySQL 怎么配合？** —— MySQL 存储为主，ES 负责搜索，双写/MQ 同步。
- **JWT 为什么放网关校验？** —— 统一鉴权，下游服务无需重复校验，透传 userId。

## 十二、风险与降级预案

| 风险 | 预案 |
|------|------|
| 微服务整合卡住 | 保留单体可运行版本兜底演示 |
| ES 部署复杂 | 先用 MySQL `LIKE` 模糊搜索顶上，ES 作为加分项 |
| 时间不够 | 优先级：核心业务 > 微服务 > Redis > MQ > ES > 压测 |
| AI 生成代码跑不通 | 分小步生成、每步运行验证，不要一次生成一大坨 |
