## 二、答辩宏观演进方向与考核要点（做的目标方向）

为了在 Day7 下午及 Day8 的项目答辩中拿到高分，项目汇报不应只是“展示一个二手交易页面”，而必须**以技术演进为主线，突出在分布式与高并发场景下的设计权衡**。

以下是答辩中应着重汇报的核心目标方向与理论支撑点：

### 1. 微服务拆分与 CAP 定理的权衡 (Day3 核心)
* **汇报要点**：
  - 项目按业务边界拆分为用户、商品、订单和网关微服务，实现了核心域的解耦。
  - **CAP 选型**：网关与各微服务接入 Nacos 注册中心。在答辩中，应明确说明 Nacos 在本项目中采用了 **AP 模式（可用性 + 分区容错性）**。因为服务注册中心首要保障在网络分区抖动时，各个健康微服务依然能够相互发现并提供服务，允许短时间内数据有轻微不一致。
  - **跨服务调用设计**：遵循“物理库独立”原则，跨服务关联一律使用 OpenFeign（如订单调用商品服务查询商品、扣减库存）。

### 2. 高并发下的详情读性能优化：Redis 旁路缓存 (Day4 核心)
* **汇报要点**：
  - 商品详情是典型的高频“读”场景。为了保护数据库免受瞬时流量冲击，引入 Redis Caching，并重点阐述对“缓存三兄弟”的解决策略：
    * **防穿透**：针对黑客恶意查询不存在的商品，缓存空对象 `NullValueMarker` 并设置 60s 极短过期时间。
    * **防击穿**：当热门商品缓存突然失效时，使用分布式互斥锁（`SETNX`）确保仅有一个线程去查库重建缓存，其他线程等待轮询，避免 DB 宕机。
    * **防雪崩**：详情缓存过期时间附加 `0~5分钟` 的随机抖动偏移，防止大批缓存同一时刻集体失效。
    * **缓存预热**：通过后台启动 Runner 主动将浏览量 Top N 的热门商品提前刷入 Redis。

### 3. 高并发下的写性能优化：RabbitMQ 异步解耦与削峰 (Day6 核心)
* **汇报要点**：
  - **异步解耦（常规交易）**：买家下单后，订单服务不直接调用通知微服务，而是向 RabbitMQ 投递 `order.notify` 消息。由消费者异步消费，向卖家推送“商品被拍下”的卡片通知。这实现了交易主流程与通知副流程的解耦，减少了下单接口的响应时延。
  - **流量削峰（限时抢购秒杀）**：
    * 答辩时重点展示：如何将写压力转移至 Redis 内存。先在 Redis 中进行原子库存扣减（`DECR`），扣减成功的请求直接投递到 RabbitMQ 的 `seckill.queue`，立即给用户响应“排队中”。
    * 后台消费者以数据库可承受的均匀速度（如 100tps）拉取消息并异步创建订单，保护 MySQL 数据库不被瞬间的流量洪峰压垮。
  - **JMeter 压测验证对比**：使用 [campus_trade_performance.jmx](file:///Users/katisarrow/summer/campus_trade_performance.jmx) 脚本，现场展示在有缓存/无缓存、有削峰/无削峰情况下的吞吐量（TPS）与响应时间（RT）对比，以真实数据说服评委。

### 4. 搜索引擎分词检索的落地：Elasticsearch (Day7 核心)
* **汇报要点**：
  - 当二手商品数据量极大时，使用 MySQL `LIKE` 模糊查询会导致索引失效且检索效率极低，且无法支持拼写容错和相关性打分。
  - 项目集成 Elasticsearch，实现了商品标题与描述的全文分词检索（IK 分词器）。
  - **双写一致性说明**：发布/修改/下架商品时采取双写策略，同步更新或删除 ES 索引，确保搜索与数据库信息的最终一致性。

### 5. 容器化一键部署与路由治理：Docker & Nginx (Day4 核心)
* **汇报要点**：
  - 传统手动配置多台虚拟机环境极其繁琐，项目采用 Docker + [docker-compose.yml](file:///Users/katisarrow/summer/docker-compose.yml) 方案，实现一键拉起包括 Nacos、Redis、RabbitMQ、ES 以及 5 大自定义业务应用在内的全栈分布式拓扑环境。
  - 前端通过 Nginx 托管静态资源，并利用 Nginx 的反向代理解决 `/api` 跨域与统一路由分发问题。
