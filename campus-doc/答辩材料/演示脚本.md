# 演示脚本

## 准备

1. 执行后端测试：`cd campus-trade && mvn test`。
2. 执行前端测试和构建：`cd campus-trade-web && npm test && npm run build`。
3. 生成后端 jar：`cd campus-trade && mvn package -DskipTests`。
4. 启动全栈：`cd /Users/katisarrow/summer && docker compose up -d --build`。
5. 打开前端页面：`http://localhost`。

## 主流程

1. 注册新用户。
   - 说明：注册走 `POST /api/user/register`，密码使用 BCrypt 哈希落库。
2. 登录。
   - 说明：登录走 `POST /api/user/login`，返回 JWT；前端拦截器后续自动携带 `Authorization`。
3. 发布商品。
   - 说明：发布走 `POST /api/product`；商品写入 MySQL，同时同步 ES，详情缓存按需重建。
4. 首页搜索商品。
   - 说明：无关键词时走 MySQL 分页列表；有关键词、价格或排序时走 ES 搜索，ES 不可用时降级到 DB 模糊查询。
5. 进入商品详情。
   - 说明：详情走 Redis cache-aside；空值缓存防穿透、互斥锁防击穿、随机 TTL 防雪崩。
6. 切换另一个账号购买商品。
   - 说明：下单走 `POST /api/order`；订单服务通过 Feign 校验商品、扣减库存、写入订单。
7. 查看卖家通知。
   - 说明：下单成功后消息进入 RabbitMQ，消费者写入 `t_notification`，前端轮询未读数。
8. 展示订单状态流转。
   - 说明：支付、确认收货、取消均使用 POST 动作接口，满足只允许 GET/POST 的交付约束。

## 秒杀说明

1. 点击商品详情页的限时秒杀。
2. 入口先预热商品和库存到 Redis。
3. `seckill:result:{productId}:{buyerId}` 防止重复提交。
4. Redis `DECR` 预扣库存，库存不足时立即回滚并返回。
5. 请求进入 RabbitMQ，消费者匀速创建订单并更新秒杀结果。

## 高频问题答案

### 为什么使用网关统一鉴权

网关是唯一可信入口，会先移除客户端伪造的 `X-User-Id`，再校验 JWT，把解析出的真实用户 ID 注入下游服务。下游只读网关注入的身份头，减少重复鉴权代码和越权风险。

### 为什么订单服务不跨表 join

微服务之间保持数据边界。订单表冗余商品标题、成交价和买卖双方 ID；需要昵称时通过用户服务批量查询，避免订单服务直接依赖用户表结构。

### Redis 缓存三类问题怎么处理

缓存穿透用空值标记，缓存击穿用互斥锁重建，缓存雪崩用随机 TTL 错峰过期。

### RabbitMQ 的作用是什么

普通下单通知不阻塞主链路；秒杀请求先进入队列削峰，由消费者按处理能力落库。

### 为什么只用 GET 和 POST

交付规则要求只允许 GET/POST。查询统一使用 GET，创建、修改、删除、状态流转等动作统一使用 POST，并由自动化测试阻止 PUT、DELETE、PATCH 回归。
