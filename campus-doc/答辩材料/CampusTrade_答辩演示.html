07

接口约束：只用 GET 和 POST

查询类接口保持 GET，创建、修改、删除和状态动作通过明确命名的 POST 路径表达语义。

GET 查询类

  • /api/product/list 表示商品分页列表。
  • /api/product/search 表示商品检索。
  • /api/order/{id} 表示订单详情。
  • /api/order/notification/list 表示通知列表。

POST 动作类

  • /api/product/{id}/update 表示商品修改。
  • /api/product/{id}/delete 表示逻辑下架。
  • /api/order/{id}/pay 表示支付状态流转。
  • /api/notification/{id}/read 表示已读状态变更。

后端门禁

Controller 反射测试阻止 PUT、DELETE、PATCH 注解回归。

前端门禁

前端测试扫描 src/api 请求方法，非 GET/POST 写法会失败。

正常路径

安全和风险：证据与边界

已经处理的风险以代码证据为准，生产化增强项以边界说明为准。

已经处理

  • 网关无条件移除客户端 X-User-Id。
  • JWT 解析失败返回统一 401 JSON。
  • 扣库存 SQL 带 status=1 AND stock>0 条件。
  • restoreStock 通过 orderNo 幂等保护。
  • 消费者重试和 DLQ 保留异常消息。

后续增强

  • 内部接口适合增加内部密钥或网段限制。
  • JWT 密钥适合从环境变量或 Secret 注入。
  • 前端 token 存储方式适合生产化调整。
  • ES 最终一致适合通过定时重建或同步链路收敛。

18

高频追问：短回答有细节

回答保持简短，但每个答案都落到项目实现。

为什么不用跨库 join

订单服务只拥有订单数据，商品标题与成交价是交易快照，用户昵称通过 Feign 批量查询。

缓存三类问题怎么落地

穿透用空值标记，击穿用互斥锁重建，雪崩用随机 TTL，Redis 异常存在本地短缓存。

MQ 的价值是什么

普通下单把通知异步化，秒杀把峰值请求排队化，用户请求不被后续处理拖住。

为什么只用 GET 和 POST

交付规则要求只允许 GET 和 POST。查询为 GET，状态动作通过明确命名的 POST 路径表达。
